# Reservation Form: Validation, Error Styling, Spacing & Consistency Fixes

## Issues to Address

1. **Validation doesn't fire as expected / hangs when switching add↔edit**
2. **Field-level error styling missing on several components**
3. **Label-to-field and error-to-field spacing needs normalization at component level**
4. **Client (Combobox) field styling inconsistent with Auto (Select)**

---

## Root Cause Analysis

### Issue 1: Validation Problems

**1a. ReservationForm: `startDate`/`endDate` not registered as FormField**
- `src/components/ReservationForm.tsx:156-164` — Dates use a plain `<FormItem>` + `<DateRangePicker>`, NOT a `<FormField>`. The zod schema validates `startDate` and `endDate`, but since they're not FormField-wrapped, `form.handleSubmit()` validation only catches them if the DateRangePicker's `onChange` triggers `setValue` with `shouldValidate: true`. The `handleDatesChange` does call `setValue('startDate', ..., { shouldValidate: true })`, so validation does fire on date selection. However, on initial submit with default dates (today→today), the `endDate > startDate` refine fails silently because both dates are the same Date object — the refine at `reservationSchema.ts:24` fires but the error is on `endDate` which has no FormField to display it.
- **Fix**: Wrap dates in a FormField for `endDate` (the refine target) so the error message displays. Keep the DateRangePicker as the control but register `endDate` as the FormField name.

**1b. Edit drawer: `endDate` not a separate FormField**
- `src/features/reservations/reservations-edit-drawer.tsx:337-349` — Only `startDate` is a FormField. The `endDate` refine error has no FormField to display it.
- **Fix**: Add a separate FormField for `endDate` that shares the same DateRangePicker control.

**1c. totalPrice auto-calc doesn't trigger validation**
- When `setValue('totalPrice', ...)` is called in the `useEffect`, it doesn't pass `{ shouldValidate: true }`. So if totalPrice was previously invalid (0), the auto-calc sets it to a valid value but the error message persists until next manual validation.
- **Fix**: Add `{ shouldValidate: true }` to totalPrice setValue calls in both forms.

**1d. Switching add→edit→create causes stale validation state**
- When `reset()` is called on mode change, previous error state may persist visually. The `reset()` function from react-hook-form clears errors, but if the `useEffect` for totalPrice runs before reset completes, it can set stale values.
- **Fix**: The `useEffect` that resets form already handles this correctly by depending on `[open, mode, reservation, reset]`. No change needed — the real issue is that `reset()` with new defaultValues clears validation errors, which is correct behavior.

**1e. `pickupLocationOther` / `dropoffLocationOther` not marked required in UI**
- Schema has `.refine()` requiring these when location is 'Other', but the form labels don't show required asterisks.
- **Fix**: Add `required` prop to these FormLabels.

### Issue 2: Missing Field-Level Error Styling

**2a. Combobox (Client in edit drawer) doesn't show error border**
- `src/components/ui/combobox.tsx:68` — Button has `aria-invalid={error}` but `error` prop is never passed from the FormField. The `FormControl` Slot sets `aria-invalid={!!error}` on the wrapper, but the Combobox Button is a child of the Slot and doesn't inherit it.
- **Fix**: Add `aria-invalid:border-destructive aria-invalid:focus-visible:ring-destructive` to the Combobox Button's className. The `FormControl` already sets `aria-invalid` on the Slot, and the Button is a direct child — but since the Button is inside a `<PopoverTrigger asChild>`, the Slot wraps the Button directly, so `aria-invalid` will cascade.

**2b. Edit drawer: raw `Select` for pickup/dropoff locations lacks error styling**
- `src/features/reservations/reservations-edit-drawer.tsx:460-469, 518-527` — Uses raw shadcn `<Select>` + `<SelectTrigger>` instead of the custom `<Select>` wrapper. The raw SelectTrigger already has `aria-invalid:border-destructive` from `select.tsx:20`, so it SHOULD show error styling via the Slot's `aria-invalid`. This actually works already — no fix needed.

**2c. Edit drawer: dailyRate/totalPrice Inputs lack error styling**
- `src/features/reservations/reservations-edit-drawer.tsx:407-414, 432-439` — Plain `<Input>` components. The base `Input` component (`input.tsx:12`) already has `aria-invalid:border-destructive`. Since FormControl sets `aria-invalid` on the Slot, and Input is a direct child, this should work. **No fix needed.**

**2d. Edit drawer: pickupLocationOther/dropoffLocationOther Inputs lack error styling**
- Same as 2c — base Input already has `aria-invalid` styling. **No fix needed.**

**2e. ReservationForm: Select wrapper doesn't pass error state**
- `src/components/Select.tsx:59` — The custom `Select` wrapper accepts `error` prop and applies `border-destructive` to SelectTrigger, but the FormField render functions never pass `error` — they only destructure `field`, not `fieldState`. However, the raw SelectTrigger in `select.tsx:20` already has `aria-invalid:border-destructive`, so the error styling works via the Slot. The custom `Select` wrapper's `error` prop is redundant but harmless. **No fix needed for functionality, but the custom Select's error handling is dead code.**

**2f. TimePicker error prop never passed**
- `src/components/ui/time-picker.tsx:10` — Accepts `error` prop, applies `border-destructive` to Input. But FormField never passes it. The base Input already has `aria-invalid:border-destructive`, so error styling works via Slot. **No fix needed for functionality.**

### Issue 3: Spacing Normalization

**3a. TimePicker has internal `space-y-2` that doubles FormItem spacing**
- `src/components/ui/time-picker.tsx:26` — `<div className={cn("space-y-2", className)}>` wraps the entire TimePicker content. When used inside a FormItem (which has `space-y-2`), the label gets double-spaced from the input.
- **Fix**: Remove `space-y-2` from TimePicker's outer div. The component should only be responsible for the input + icon layout. Label and error message should be handled by FormItem/FormLabel/FormMessage at the usage site.

**3b. TimePicker renders its own label and error — conflicts with FormField pattern**
- `src/components/ui/time-picker.tsx:27-31` — Renders `<Label>` internally. But in both forms, the TimePicker is used inside a FormField that already provides FormLabel and FormMessage. The internal label renders a SECOND label.
- **Fix**: Remove the internal Label and error rendering from TimePicker. Make it just the input + icon. Let FormField/FormLabel/FormMessage handle label and error at the usage site. Pass `error` prop only for the Input's className (or rely on `aria-invalid`).

**3c. Custom Select wrapper has `gap-0.5` — inconsistent with FormItem `space-y-2`**
- `src/components/Select.tsx:52` — `gap-0.5` between label and select. This is used in ReservationForm for Client/Auto/Vehicle. The FormItem provides `space-y-2` (0.5rem), and the Select wrapper adds `gap-0.5` (0.125rem) internally. Since the Select wrapper renders its OWN label (not using FormLabel), the FormItem's `space-y-2` controls spacing between FormLabel and the Select wrapper, and the Select's internal `gap-0.5` controls spacing between its internal label and the select trigger. This is actually fine — the internal label is a different element.
- **Wait**: Looking more carefully at `Select.tsx:52-71` — when `options` are provided, the Select wrapper renders its OWN `<Label>` AND the FormField also renders a `<FormLabel>`. This means there are TWO labels for the same field. The FormLabel is outside the Select div, and the Select's internal Label is inside. This creates visual duplication.
- **Fix**: The custom `Select` wrapper should NOT render its own Label when used inside a FormField. Remove the internal Label rendering from the `options` branch. The label should only come from FormLabel.

**3d. Combobox has no internal spacing — relies entirely on FormItem**
- This is correct. No fix needed.

### Issue 4: Client (Combobox) vs Auto (Select) Styling Inconsistency

**4a. Combobox placeholder uses `<span data-placeholder>` instead of `data-[placeholder]` selector**
- `src/components/ui/combobox.tsx:76` — Placeholder is rendered as `<span data-placeholder className="text-muted-foreground">{placeholder}</span>`. This is a custom approach, not the standard `data-[placeholder]:text-muted-foreground` used by SelectTrigger.
- **Fix**: Change Combobox to use `data-[placeholder]:text-muted-foreground` on the Button, matching the SelectTrigger pattern. Remove the `<span data-placeholder>` wrapper.

**4b. Combobox Button lacks `min-h-9` — may have different height than SelectTrigger**
- `src/components/ui/combobox.tsx:69-73` — Button className doesn't include `min-h-9`. SelectTrigger has `min-h-9`.
- **Fix**: Add `min-h-9` to Combobox Button className.

**4c. Combobox Button lacks `aria-invalid` styling**
- Already covered in 2a. Add `aria-invalid:border-destructive aria-invalid:focus-visible:ring-destructive`.

Actually, the cleanest approach for both forms is to keep the current DateRangePicker pattern but add a separate `FormField` for `endDate` that doesn't render its own control — it just captures the error. But react-hook-form's FormField requires a render function that returns a control.

**Revised approach**: Register `endDate` as a hidden field that shares the same DateRangePicker logic:

```typescript
// In both forms, add alongside the DateRangePicker FormItem:
<FormField
  control={form.control}
  name="endDate"
  render={() => (
    <FormItem className="sr-only">
      <FormMessage />
    </FormItem>
  )}
/>
```

This renders only the FormMessage for endDate errors, without duplicating the control. The `startDate` FormField (or the plain FormItem in ReservationForm) handles the DateRangePicker display.
