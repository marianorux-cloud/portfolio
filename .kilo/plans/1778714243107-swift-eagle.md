# Credit Card Storage Plan

## Context
- Project: auto-lola (React/TypeScript/Vite frontend)
- Goal: Add encrypted credit card storage to `Renter` type
- Current state: **Frontend-only with localStorage** - NO backend detected
- Data storage: Uses localStorage for autos, renters, reservations
- **Clarification**: No checkout flow - cards for client records management

## Current Data Model
```typescript
// Renters are stored in localStorage with these fields:
interface Renter {
  id: string;
  name: string;
  phone: string;
  notes?: string;
}
```
New field needed: encrypted credit card data

## Critical Finding
This is a **pure frontend app** with no backend. All data is stored in browser localStorage. This fundamentally changes the options.

## Options Analysis

### Option A: Stripe SetupIntents (Recommended)
**PCI Compliance**: SAQ-A (simplest)
**Implementation**:
1. Add backend (or use Stripe Edge Functions)
2. Install: `npm install @stripe/react-stripe-js @stripe/stripe-js`
3. Use `CardElement` to collect card client-side
4. Backend stores `PaymentMethod` ID in localStorage or database

**Pros**:
- Card data never touches your server
- Stripe handles PCI compliance
- Supports future/recurring payments

**Cons**:
- Requires architectural change (add backend)

### Option B: Browser Crypto + localStorage (ONLY viable for frontend-only)
**PCI Compliance**: SAQ-D (hardest)
**Implementation**:
1. Encrypt card in browser: `window.crypto.subtle.encrypt()`
2. Store encrypted data + key in localStorage
3. Decrypt when needed

**Pros**:
- No backend required
- Data stays client-side

**Cons**:
- **YOU ARE NOW RESPONSIBLE FOR PCI COMPLIANCE**
- Key must be stored somewhere (in localStorage = not secure)
- SAQ-D requires extensive security audits
- Most projects abandon this for Stripe

### Option C: Third-Party Client-Side Vault
**PCI Compliance**: Varies
**Services**: 
- Stripe SetupIntents (requires backend)
- Braintree Vault (requires backend)

### Option D: No Persistent Storage
**PCI Compliance**: None
**Implementation**:
- Collect card in memory during session
- Not persisted between browser sessions

## Recommendation
**For a frontend-only app**: 

1. **If you have a backend planned**: Use **Stripe SetupIntents** - industry standard
2. **If no backend**: Consider **no persistent storage** or add a backend first

The browser crypto approach puts you at SAQ-D which is significantly more complex than it's worth.

## Questions
1. **Is a backend planned?** This determines if Stripe SetupIntents is viable.
2. **Do you need cards persisted across sessions?** If not, use memory-only storage.
3. **What's the use case?** Understanding why cards need storage helps determine the best approach.
